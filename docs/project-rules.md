# Project Rules

MiniNet 是一个学习项目，核心目的有两个：

1. 学习游戏网络传输层的基础设计与实现。
2. 熟悉和 AI agent 协作完成软件项目的流程。

因此，本项目的成功标准不是“最快写完最多功能”，而是“每一步都能解释、能测试、能 review、能复盘”。

## 基本原则

- 小步提交：每个 issue 只解决一个清晰问题。
- 学习优先：代码要能被初学者读懂，避免过早抽象。
- 可验证优先：每个功能都要有验收标准和测试思路。
- 保守实现：优先使用简单、稳定、标准库可支持的方案。
- 明确范围：每个 issue 都要写清楚包含什么、不包含什么。
- 不追求生产级：暂不实现加密、NAT 打洞、拥塞控制、分片、多频道优先级等复杂能力。

## 技术规则

- 默认语言：C++。
- 默认构建：CMake。
- 默认平台：Windows。
- 网络基础：UDP。
- 测试优先级：单元测试优先，其次是本地手动 demo。
- 第三方依赖：默认不引入，确实需要时必须在 issue 或 PR 中说明原因。
- 日志：网络行为要尽量可观察，例如连接、心跳、ACK、重传、丢包模拟。
- 配置：延迟、丢包率、超时时间等参数应可配置，不要散落在代码里。

## 代码注释规则

MiniNet 是学习项目，注释要服务于理解网络和 C++ 实现，而不是只重复代码表面含义。

头文件 `.hpp` 的注释要求：

- 每个常量要说明用途、单位或协议含义。
- 每个 enum 要说明它表达的状态或分类。
- 每个 struct / class 要说明它在协议或程序结构中的角色。
- 每个 public 函数要说明调用者为什么需要它、输入代表什么、返回值代表什么。
- 每个重要成员变量要说明它保存什么状态。
- 如果某个类型是为了兼容当前工具链或 Windows 环境，也要说明原因。

实现文件 `.cpp` 的注释要求：

- 每个非平凡函数要说明整体流程和关键设计选择。
- 涉及网络、字节序、资源释放、超时、错误处理、协议校验的代码必须解释原因。
- 简单 getter、字符串转换、明显的构造函数可以少写或不写注释。
- 注释优先解释“为什么这样做”，不要只把代码翻译成中文。
- 如果当前实现是临时学习版或刻意简化，要明确写出限制，方便后续 issue 扩展。

## 网络实现边界

项目会逐步实现：

- UDP Ping/Pong
- 逻辑连接
- 心跳与超时
- 包序号
- ACK 与 ACK bits
- 可靠消息
- 丢包重传
- 不可靠状态同步
- 延迟、丢包、乱序、重复包模拟

项目暂时不实现：

- 加密
- 压缩
- NAT 打洞
- 分片与大包传输
- 拥塞控制
- 完整 RPC 框架
- 生产级安全校验
- Unity 集成

## Issue 规则

每个 issue 应该尽量包含：

- 背景：当前状态是什么，为什么要做。
- 目标：这次要完成什么。
- 范围：包含什么、不包含什么。
- 验收标准：完成后如何判断通过。
- 测试用例：至少列出正常路径和一个边界情况。
- 备注：实现建议、风险或后续 issue。

Issue 不应该太大。如果一个 issue 需要同时改协议、状态机、重传和 demo，应该拆分。

## 验收标准规则

验收标准要写成可判断的句子，避免只写“实现某功能”。

推荐写法：

```md
- 客户端发送 Ping 后，服务端返回 Pong。
- 客户端收到 Pong 后打印 RTT。
- 服务端收到非法数据包时不会崩溃。
```

不推荐写法：

```md
- 实现 Ping/Pong。
- 优化网络。
- 代码写好。
```

## 测试用例规则

每个功能至少考虑：

- 正常路径：功能按预期工作。
- 异常输入：非法包、未知包、空数据等。
- 超时或失败：服务端不存在、心跳丢失、ACK 未返回等。
- 重复或乱序：适用于序号、ACK、重传相关功能。

如果某个功能暂时没有自动化测试，需要在 PR 中说明手动验证方式。

## PR 规则

每个 PR 应该说明：

- Summary：这个 PR 做了什么。
- Changes：主要改动列表。
- Tests：运行了哪些测试，或为什么没测。
- Acceptance Criteria：对应 issue 的验收标准是否完成。
- Review Focus：希望 reviewer 重点看什么。
- Notes：刻意没做的内容、已知限制和后续 issue。

PR 描述要诚实，不要把未实现的内容写成已完成。

## Review 规则

Review 重点看：

- 是否满足 issue 的验收标准。
- 是否有明显边界情况遗漏。
- 是否有职责混乱，例如解析、状态修改、发送响应全部写在一个大方法里。
- 是否有测试或手动验证说明。
- 是否引入了不必要的复杂度。
- 是否影响后续阶段扩展。

好的 review comment 应该具体、可执行，并说明原因。

示例：

```md
这里收到重复 ClientHello 会创建新的 connection id。
如果客户端因为丢包重发握手，服务端可能产生重复连接。
建议根据 endpoint 做去重，或者在当前 issue 中明确不处理重复握手。
```

## AI Agent 协作规则

使用 AI agent 时，尽量遵循：

- 先给 issue，再让 AI 实现。
- 开始 issue 前先判断规模：`small`、`normal`、`large`。
- `small` 修改使用轻量流程，不启动完整 subagent。
- `normal` 功能 issue 使用精简四 agent 工作流：`issue_reviewer -> architecture_planner -> implementation_worker -> test_worker`。
- `large` 或高风险 issue 使用完整四 agent 工作流。
- 让 AI 说明计划，但不要一次性实现太大范围。
- 要求 AI 保持改动小而可 review。
- 要求 AI 运行测试或说明为什么不能运行。
- 对 AI 生成代码进行人工 review。
- 不直接接受看不懂的实现，先要求解释。

Token 使用规则：

- 小型文档、注释、模板、README 修改，不启动完整 subagent 工作流。
- 普通功能 issue 默认使用精简四 agent 工作流，只给每个 agent 当前 issue 必要上下文。
- 只有涉及协议设计变化、跨模块重构、复杂状态机或你明确要求时，才使用完整工作流。
- 每个 agent 输出优先给结论、修改点、风险和测试结果，避免重复项目背景。
- `architecture_planner` 的输出必须保存到 `docs/architecture-plans/`，方便学习和复盘。

AI 可以帮助：

- 拆 issue。
- 补验收标准。
- 补测试用例。
- 实现代码。
- 写 PR 描述。
- 做 code review。
- 解释协议设计和代码。

AI 不应该替代：

- 你对项目目标的判断。
- 你对需求边界的确认。
- 你对最终代码的理解。
- 你对是否合并 PR 的决定。

详细流程见 [agent-workflow.md](agent-workflow.md)。

## 提交规则

- 提交信息使用简短英文，便于 GitHub 展示。
- 常用前缀：
  - `docs:` 文档变更
  - `chore:` 项目结构、配置、模板
  - `feat:` 新功能
  - `fix:` 修复问题
  - `test:` 测试相关
  - `refactor:` 不改变行为的重构
- 每次提交尽量对应一个清晰目的。

示例：

```text
docs: add project rules
feat: add UDP ping pong
test: cover malformed packet handling
```

## 学习复盘规则

每完成一个阶段，建议记录：

- 这次学到的网络概念。
- AI 哪些地方帮得上忙。
- AI 哪些地方需要你纠正。
- issue 是否写得足够清楚。
- 验收标准是否真的可验证。
- 下一个 issue 应该如何拆得更好。

这部分可以写在 issue comment、PR comment，或后续单独整理到文档中。
